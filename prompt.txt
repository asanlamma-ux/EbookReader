‚ÄéExecutive Summary
‚ÄéMiyo is a thoughtfully built EPUB reader specifically designed for the web-novel and Machine Translation (MTL) reading community ‚Äî a niche that mainstream EPUB readers have almost entirely ignored. Its core differentiators ‚Äî term replacement/correction, inline translation, ambient particle-effect themes, and bionic reading ‚Äî are genuinely unique in the market and align with what this audience actually needs.
‚Äé
‚ÄéHowever, the app carries several meaningful risks: a 3,445-line monolithic reader file that creates memory and orientation bugs; a WebView-based EPUB renderer with real XSS exposure from malicious EPUB files; unencrypted local storage of sensitive tokens; and Supabase credentials embedded in the app bundle without verifiable Row-Level Security enforcement.
‚Äé
‚ÄéOn the feature side, the single most requested capability across web-novel communities that Miyo is missing is Text-to-Speech (TTS) with reading sync, followed by vertical/continuous scroll mode and AMOLED black theme. Competitors like Moon+ Reader and KOReader cover these basics while lacking everything Miyo does well, meaning Miyo is well-positioned ‚Äî but needs to close these gaps to convert power users.
‚Äé
‚ÄéThe App's Strongest Features
‚ÄéThese are Miyo's genuine competitive moats ‚Äî things no mainstream competitor offers:
‚Äé
‚Äé1. Term Group System (MTL Correction Engine) ‚Äî The ability to create named groups of find-and-replace rules that auto-apply during rendering is the single biggest differentiator. No other mainstream reader has this. Reddit's MTL community considers this a "dream feature" and it's executed well in the codebase with proper word-boundary detection for Latin terms.
‚Äé
‚Äé2. Ambient Particle Themes ‚Äî The "Peach Blossom," "Dark Coffee," and other art-pack themes with particle effects are a genuine delight and a clear signal of design investment. No competitor even attempts this.
‚Äé
‚Äé3. Inline Chapter Translation ‚Äî Translating entire chapters inline without leaving the app is rare. Most apps only offer tap-to-translate for selections. This is a major advantage for Chinese/Japanese MTL readers.
‚Äé
‚Äé4. Bionic Reading Mode ‚Äî Present in very few competitors. The implementation here appears solid.
‚Äé
‚Äé5. WTR-LAB Integration + OPDS ‚Äî An in-app browser wired to an MTL novel catalog, combined with OPDS for Project Gutenberg access, gives Miyo a content discovery layer that pure EPUB readers lack entirely.

‚ÄéVulnerabilities & Security Risks
‚ÄéCritical: WebView XSS via EPUB Content Injection
‚ÄéThe reader renders EPUB chapters by injecting HTML content into a WebView. EPUB files are ZIP archives and can contain arbitrary HTML, CSS, and JavaScript. A malicious EPUB can embed scripts that, depending on WebView configuration, may execute with access to the local filesystem.
‚Äé
‚ÄéThe codebase does have a url-safety.ts module and an escapeHtmlContent/escapeHtmlAttribute pair in the reader ‚Äî these are good. However, the risk is that full chapter HTML (from the EPUB itself) is injected wholesale into the WebView document, not escaped. If the WebView's allowFileAccess setting is enabled, a crafted EPUB chapter could use fetch or XMLHttpRequest to read internal app storage. Research confirms this is a known class of Android WebView vulnerability (CVE-2020-6506) and is particularly acute for file-based content.
‚Äé
‚ÄéWhat to do: Verify allowFileAccess, allowFileAccessFromFileURLs, and allowUniversalAccessFromFileURLs are explicitly set to false in ReaderWebView. Add a Content Security Policy header to the injected HTML document that blocks all external resource loads and inline scripts.
‚Äé
‚ÄéHigh: Unencrypted Local Storage of Sensitive Data
‚ÄéBoth AsyncStorage and react-native-mmkv are used throughout the app. Neither encrypts data by default. On a rooted Android device or via ADB backup, all tokens, reading progress, and potentially cached Supabase session data stored in these are readable in plaintext. The security research specifically flagged MMKV as unencrypted unless an encryption key is explicitly provided and managed via Android Keystore / iOS Keychain ‚Äî there's no evidence in the codebase that this is being done.
‚Äé
‚ÄéHigh: Supabase Anon Key in App Bundle + RLS Dependency
‚ÄéThe Supabase anon key is embedded in the app bundle via environment variables, which is standard practice ‚Äî but it means the entire security model depends on Row-Level Security (RLS) policies being correctly configured on every table. If any table is missing an RLS policy or has an overly permissive one (e.g., authenticated role can read all rows regardless of user_id), any user can enumerate other users' data using the public API. This cannot be verified from the client codebase alone but is a critical configuration risk.
‚Äé
‚ÄéModerate: HTTP Allowed by Default in URL Safety
‚ÄéisSafeExternalUrl in url-safety.ts has allowHttp: true as the default. This means the app will follow HTTP (unencrypted) links, exposing users on shared networks to MITM attacks when the WebView navigates to external URLs.
‚Äé
‚ÄéLikely Bugs & Glitches
‚Äé1. Orientation Change Breaks Layout
‚ÄéDimensions.get('window') is called at module level in app/reader/[id].tsx (line 106) and stored in screenWidth/screenHeight constants. These values never update when the device rotates. This means the reader layout will be wrong after any orientation change until the user navigates away and back. The fix is to use the useWindowDimensions hook inside the component, as the Home screen correctly does.
‚Äé
‚Äé2. The Reader File Is 3,445 Lines ‚Äî Memory Leaks Are Likely
‚ÄéThis single file manages the WebView, all panels (typography, themes, search, annotations, translation, dictionary, stats), all state, all gestures, and all JS injection. At this scale, it's nearly certain there are stale closure bugs, effect cleanup misses, and memory leaks ‚Äî especially around the WebView instance and interval timers. Long reading sessions will gradually degrade performance.
‚Äé
‚Äé3. Reading Session Stats Can Be Corrupted by Concurrent Sessions
‚ÄésessionStartTime, sessionBookId, sessionWordsRead, and sessionChaptersRead in reading-stats.ts are module-level global variables (lines 40‚Äì43). If a user opens two books in quick succession (or a background notification triggers a session start), these globals are silently overwritten and the previous session's data is lost. This will manifest as incorrect reading streaks and word counts.
‚Äé
‚Äé4. Chapter Translation Fails Silently on Long Chapters
‚Äéinline-translate.ts sets MAX_CHARS = 440 for the free MyMemory API. This is fine for short text selections, but full-chapter translation would require chunking long HTML content into many sequential API calls with rate-limit handling. If this chunking logic doesn't exist, long chapters will be silently truncated or partially translated, with no user-visible error explaining why.
‚Äé
‚Äé5. Translation Cache Uses AsyncStorage (Slow)
‚ÄéChapter translation results are cached using AsyncStorage with prefix @miyo/chapter-translation/. For a reader app with large chunks of HTML, AsyncStorage's synchronous-bridge architecture will create noticeable lag when loading cached translations. This should use MMKV for I/O speed.
‚Äé
‚Äé6. endReadingSession Min-Duration Filter May Eat Short Intentional Sessions
‚ÄéThe session recorder filters out sessions under 30 seconds (seen from line 80 onward). For a user who opens a book, reads one chapter quickly, and closes ‚Äî a fast reader could genuinely finish a short chapter in under 30 seconds. Their session vanishes from stats, breaking streak tracking.
‚Äé
‚Äé7. Google OAuth Disabled via Beta Flag ‚Äî No User-Visible Explanation
‚ÄéAuthContext imports isGoogleSignInDisabled from beta-flags.ts. If this flag is active, the "Sign in with Google" button presumably disappears or fails silently. Users who try to use it and get no feedback will abandon account creation.
‚Äé
‚Äé8. HTTP Default in URL Safety (Doubles as Bug)
‚ÄéExternal link navigation defaults to allowing plain HTTP. On iOS, Apple's ATS (App Transport Security) may block these requests entirely with no user-facing error message ‚Äî the link simply does nothing, which feels like a bug.

‚ÄéCompetitive Gaps ‚Äî What Miyo Is Missing
‚ÄéBased on Reddit communities (r/ebooks, r/noveltranslations, r/androidapps, NovelUpdates forums) and competitor analysis:
‚Äé
‚ÄéTier 1: Most Requested, Highest Impact
‚ÄéText-to-Speech (TTS) with Reading Sync ‚Äî The single most-requested feature in EPUB reader communities and completely absent from Miyo. Reddit users explicitly want TTS where the current word is highlighted as it's spoken ("read while commute, keep my place"). Moon+ Reader Pro offers this. For web-novel readers who consume extremely long books, this is a daily-use feature. The Expo ecosystem has expo-speech available which could provide a starting point.
‚Äé
‚ÄéVertical / Continuous Scroll Mode ‚Äî Reddit's MTL community is emphatic: "Web novel readers are used to scrolling, not paging." Miyo appears to use chapter-by-chapter page turning. A smooth infinite-scroll mode (like how most web novel websites work) would dramatically improve comfort for the core audience. This is a significant architectural change to the WebView rendering.
‚Äé
‚ÄéAMOLED True Black Theme ‚Äî Multiple Reddit threads explicitly call this out: "I need a real black background, not dark gray." Miyo's existing dark themes likely use dark gray backgrounds (#1a1a1a range) rather than pure #000000. For OLED screen users, pure black saves meaningful battery and reduces eye strain. This is a minor code change with high perceived impact.
‚Äé
‚ÄéTier 2: High Value, Moderate Effort
‚ÄéPDF Support ‚Äî Every major competitor (ReadEra, Moon+, KOReader) supports PDF. While Miyo is positioned as an EPUB-first app, users with mixed libraries will consistently hit this wall. PDF support would also unlock academic paper reading.

‚ÄéAnalysis & Strategic Synthesis
‚ÄéMiyo is operating in a real underserved niche ‚Äî MTL web novel readers on mobile are poorly served by both general EPUB readers (which ignore their translation/correction needs) and web novel aggregator apps (which are ad-ridden scrapers without quality reading environments). The positioning is correct and defensible.
‚Äé
‚ÄéThe most dangerous strategic gap is not a missing feature ‚Äî it's the 3,445-line reader file. As the app grows, every new feature bolted onto this file increases the probability of memory leaks, regression bugs, and orientation glitches that will drive away quality-conscious users. Breaking the reader into composable panels (each with its own file) is the most important architectural investment before scaling further.
‚Äé
‚ÄéOn the market side, the TTS gap is the sharpest vulnerability. Any competitor that adds MTL term replacement + TTS would immediately threaten Miyo's core audience. This should be prioritized above ambient themes and gamification.
‚Äé
‚ÄéThe security model is structurally sound (Supabase + RLS is a reasonable choice) but the implementation assumptions need auditing ‚Äî specifically, confirming WebView file access is locked down and that every Supabase table has RLS enabled with user-scoped policies.
